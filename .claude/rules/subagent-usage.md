# サブエージェントの使用ルール

## 自動的に使用すべきタイミング

| エージェント | トリガー条件 |
|-------------|-------------|
| `pre-commit-check` | コミット前のビルド・テスト検証時。手動で `pnpm build` / `pnpm test` を個別実行する代わりにこのエージェントを使う。 |
| `code-review` | 実装完了後、コミット前にレビューが必要な場合。ユーザーから「レビューして」と指示された場合。 |
| `craft-component-scaffold` | 新規 Craft コンポーネントの実装を開始する前（Planモードでの調査フェーズ）。 |
| `impact-analysis` | 既存ファイルの変更を伴うタスクのPlanモードで、影響範囲の把握が必要な場合。 |
| `moc-data-validator` | .moc ファイルのデータ構造に影響する変更を行った後、または .moc ファイルの不整合を調査する場合。 |

## 使い分けの指針

- **コミット前**: `code-review` → 指摘修正 → `pre-commit-check` → コミット の順で使う。
- **新規コンポーネント追加**: `craft-component-scaffold` で調査 → Planモードで計画 → 実装 → `code-review` の順で使う。
- **既存コード変更**: 必要に応じて `impact-analysis` でPlan精度を上げる。
- **.moc関連の変更**: `moc-data-validator` で変更前後の整合性を確認する。

## 注意事項

- サブエージェントの結果はユーザーに要約して報告すること。
- サブエージェントの指摘をすべて鵜呑みにせず、文脈に応じて判断すること。
