# 実装ワークフロー

## Planモードの使用

- コードの新規追加・変更を伴うタスクは、**必ずPlanモードで計画を立て、ユーザーの承認を得てから実装を開始**すること。
- Planモードでは以下を明示すること:
  - 参照した既存コードとその場所
  - 踏襲するパターン
  - 変更するファイルと変更内容
- ユーザーの承認なしに実装を開始してはならない。
- **Planモードで承認された実装方針を実装中に変更してはならない。** 変更が必要な場合は実装を中断しユーザーに許可を取ること。

## AskUserQuestion の使用

- 実装方針が複数考えられ、どちらを選ぶかがユーザーの意図に依存する場合は、**Planモード内でAskUserQuestionを使って確認**すること。
- 以下の場合は確認不要（自己判断で進める）:
  - 既存パターンが明確に存在し、それに従うだけの場合
  - タスク内容から意図が明確に読み取れる場合
- 以下の場合は確認必須:
  - 既存パターンが複数あり、どれを踏襲するか判断できない場合
  - 要件が曖昧で実装方針が定まらない場合
  - 既存の設計を変更・破壊する可能性がある場合
- 些細な確認を繰り返してユーザーの手間を増やさないこと。確認は本当に必要な場合のみ行う。

## 実装前の一次ソース確認

- 実装に必要な仕様・定義・値は、**必ず一次ソースを確認してから**実装すること。
- 一次ソースとは、プロジェクト内の既存コード、設定ファイル、依存ライブラリのソースコード、公式ドキュメントなど、実際の定義元を指す。
- 「知っているつもり」で値をハードコードしたり、推測でオリジナルの定義を作成してはならない。
- 自己知識やWeb検索の結果より、プロジェクト内の実ソースを常に優先すること。

## 実装前の既存パターン確認

- 新しいUIや機能を実装する前に、**必ずプロジェクト内で同種・類似の実装をGrep/Globで検索**すること。
- 既存の実装パターン・コンポーネント・データ定義が見つかった場合は、それを再利用・踏襲すること。オリジナルの実装を新規に作らない。
- 確認対象の例:
  - UIコンポーネント（カラーピッカー、スライダー、セレクタなど）
  - データ定義（カラーパレット、定数、設定値など）
  - レイアウトパターン、スタイリング手法
- この確認を怠りオリジナル実装を作ることは禁止する。
